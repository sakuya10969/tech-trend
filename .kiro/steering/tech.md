# 技術選定と方針

## フロントエンド採用技術

| 技術 | バージョン目安 | 役割 |
|------|---------------|------|
| Next.js | 16.x | アプリケーションフレームワーク（App Router 使用） |
| React | 19.x | UI ライブラリ（React Compiler 有効） |
| TypeScript | 5.x | 型安全な開発 |
| Mantine | 最新安定版 | UI コンポーネントライブラリ |
| Orval | 最新安定版 | OpenAPI からの API クライアント・型・Zod スキーマ自動生成 |
| Zod | 最新安定版 | ランタイムバリデーション（Orval 生成物として利用） |
| Biome | 2.x | リンター・フォーマッター |

## バックエンド採用技術

| 技術 | バージョン目安 | 役割 |
|------|---------------|------|
| Hono | 4.x | Web フレームワーク |
| TypeScript | ESNext | 型安全な開発 |
| Cloudflare Workers | - | 実行環境（Wrangler でデプロイ） |
| OpenAPI（Zod OpenAPI Hono） | - | API スキーマ定義・ドキュメント生成 |
| GraphQL（リクエストライブラリ） | - | GitHub API との通信 |

## パッケージマネージャ・ツールチェイン

- パッケージマネージャ: Bun
- フロントエンドリンター・フォーマッター: Biome
- バックエンドデプロイ: Wrangler

## 技術選定の理由

### Next.js + React 19

- App Router によるファイルベースルーティングと Server Components の活用
- React Compiler による自動最適化（手動メモ化の削減）
- 将来的な SSR/ISR 活用の余地を残す

### Mantine

- 高品質なコンポーネント群を提供し、UI 実装の工数を削減する
- テーマカスタマイズが柔軟で、独自デザインへの対応が容易
- アクセシビリティ対応が組み込まれている
- データ表示系コンポーネント（テーブル、チャート連携等）が充実しており、トレンド可視化と相性が良い

### Hono + Cloudflare Workers

- 軽量かつ高速な Web フレームワークで、エッジ実行環境と相性が良い
- OpenAPI 統合（Zod OpenAPI Hono）により、スキーマ定義とルーティングを一体化できる
- Cloudflare Workers はコールドスタートが極めて短く、API サーバーとして適している

### Orval

- OpenAPI 定義からクライアントコード・TypeScript 型・Zod スキーマを自動生成する
- フロントエンドとバックエンドの型の乖離を防ぎ、契約ベースの開発を実現する
- 手書きの API クライアントコードを排除し、保守コストを下げる

### Biome

- リンターとフォーマッターを単一ツールで統合し、設定の複雑さを減らす
- ESLint + Prettier の組み合わせより高速に動作する

## 技術方針

### OpenAPI を契約として扱う

- OpenAPI 定義はバックエンドが正とする（Zod OpenAPI Hono でルート定義と同時に生成）
- フロントエンドは OpenAPI 定義から Orval で自動生成したクライアント・型・スキーマのみを使用する
- OpenAPI 定義の変更はフロントエンド・バックエンド間の契約変更として扱い、双方の整合性を担保する
- 手書きの API 型定義やクライアントコードは作成しない

### フロントエンドから外部 API を直接叩かない

- フロントエンドは Hono バックエンドの REST API のみを呼び出す
- GitHub API やその他外部サービスの API をフロントエンドから直接呼び出すことは禁止する
- 理由:
  - 認証トークンをフロントエンドに露出させない
  - 外部 API の仕様変更・レート制限をバックエンドで一元管理する
  - データソース追加時にフロントエンドの変更を最小限に抑える
  - 外部 API のレスポンス形式の差異をフロントエンドに漏らさない

### 外部公開境界は REST/OpenAPI、内部統合は GraphQL

- バックエンドが外部に公開する API は REST（OpenAPI 準拠）とする
- バックエンド内部で GitHub API と通信する際は GraphQL を優先的に使用する
- 理由:
  - GitHub GraphQL API は必要なフィールドだけを取得でき、複数リソースを1リクエストで取得できる
  - REST API では複数エンドポイントへのリクエストが必要な情報も、GraphQL なら1回で取得可能
  - 一方、フロントエンドとの境界は REST/OpenAPI にすることで、Orval による自動生成や標準的なツールチェインとの互換性を維持する

### 外部 API の差分はバックエンドが吸収する

- GitHub API のレスポンス構造、認証方式、ページネーション仕様などはバックエンド内部に閉じ込める
- フロントエンドに返すデータは、プロダクト固有のスキーマに整形・集約する
- 将来 Reddit や Hacker News を追加する際も、フロントエンドの API インターフェースへの影響を最小化する
