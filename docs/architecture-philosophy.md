# アーキテクチャ思想

## 全体方針

- フロントエンドとバックエンドの責務を明確に分離する
- OpenAPI を単なるドキュメントではなく、フロントエンドとバックエンドの契約として扱う
- 外部 API の差分はバックエンドが吸収し、フロントエンドに漏らさない
- 将来のデータソース追加に耐えられる構造を前提とするが、現時点で過剰な抽象化は避ける

## フロントエンド

### Feature-Sliced Design（FSD）

- レイヤー構成: app → pages → widgets → features → entities → shared
- 上位レイヤーは下位レイヤーを参照できるが、逆方向の参照は禁止
- 同一レイヤー内のスライス間は直接参照しない
- ページから直接 API を叩く構成を避け、責務を適切なレイヤーに分散する

### UI 方針

- Mantine を UI コンポーネントの基盤とする
- プロジェクト固有のラッパーが必要な場合は shared/ui に配置する
- 各レイヤーの ui/ ディレクトリ内で Mantine コンポーネントを直接利用してよい

### API 連携方針

- フロントエンドは Hono バックエンドの REST API のみを呼び出す
- 外部サービスの API を直接呼び出すことは禁止
- Orval 生成物のみを API クライアントとして使用し、手書きの API クライアントは作成しない
- 生成コードと手書きコードの責務を明確に分離する

## バックエンド

### モジュラーモノリス

- データソース単位でモジュールを分離する（github, reddit, hackernews 等）
- 各モジュールは `server/src/` 直下にディレクトリとして配置する（`modules/` のような中間ディレクトリは設けない）
- 各モジュールは routes / usecases / clients / schemas / types の層構成を持つ
- モジュール間の直接参照は避け、公開インターフェースは routes のみ
- 共通処理は shared/ に配置する

### 外部 API 統合方針

- 外部 API との通信はモジュール内の clients/ 層に閉じ込める
- GitHub API には GraphQL を優先的に使用する（必要なフィールドのみ取得、リクエスト数削減）
- 認証トークン、ページネーション、レート制限対応は clients/ 層で吸収する
- 外部 API のレスポンスをそのまま返さず、usecases/ 層でプロダクト固有のスキーマに変換する

### API 公開方針

- 外部公開境界は REST/OpenAPI とする
- Zod OpenAPI Hono でルート定義とスキーマ定義を一体化する
- コードファーストアプローチで OpenAPI ドキュメントを自動生成する
